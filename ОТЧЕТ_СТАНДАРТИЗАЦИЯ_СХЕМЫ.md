## 4. Сравнение ECS и OCSF

### 4.1 Elastic Common Schema (ECS)

**Официальная спецификация:** https://www.elastic.co/guide/en/ecs/current/

ECS — это открытая схема Elastic для структурирования данных событий в Elasticsearch.

| Аспект | Описание |
|--------|----------|
| **Основное применение** | SIEM, аналитика логов, поиск угроз в Elastic Stack |
| **Структура** | Иерархическая точечная нотация (напр., `source.ip`) |
| **Управление** | Elastic с процессом RFC от сообщества |
| **Инструменты** | Нативная поддержка в Beats, Logstash, Elasticsearch, Kibana |

**Категории полей ECS:**
- Base, Agent, Client, Cloud, Container
- Destination, DNS, Error, Event, File
- Host, HTTP, Log, Network, Observer
- Process, Server, Source, TLS, URL, User

### 4.2 Open Cybersecurity Schema Framework (OCSF)

**Официальная спецификация:** https://schema.ocsf.io/

OCSF — вендор-нейтральная схема, разработанная консорциумом AWS, Splunk, IBM и других.

| Аспект | Описание |
|--------|----------|
| **Основное применение** | Кросс-вендорный обмен событиями, озёра данных безопасности |
| **Структура** | Иерархическая со строгой типизацией и классами событий |
| **Управление** | Консорциум Linux Foundation |
| **Инструменты** | Amazon Security Lake, растущая поддержка |

**Классы событий OCSF:**
- System Activity, Findings, IAM
- Network Activity, Discovery, Application Activity

### 4.3 Ключевые различия

| Критерий | ECS | OCSF |
|----------|-----|------|
| **Происхождение** | Один вендор (Elastic) | Консорциум вендоров |
| **Зрелость** | Зрелая, широко развёрнута | Развивающаяся |
| **Гибкость** | Расширяема через custom поля | Строгая валидация схемы |
| **Временные метки** | ISO 8601 строки | Unix миллисекунды |
| **ID стратегия** | Строковые UID | Целочисленные class/category UID |

### 4.4 Когда использовать

**Выбирайте ECS когда:**
- Развёртываете Elastic SIEM или Observability
- Используете инфраструктуру Elastic Beats/Agent
- Нужны зрелые, хорошо документированные маппинги
- Требуется совместимость с правилами детекции Elastic

**Выбирайте OCSF когда:**
- Строите вендор-нейтральные озёра данных безопасности
- Используете Amazon Security Lake
- Требуется строгая валидация схемы
- Интегрируете множество вендоров безопасности

### 4.5 Выбор для данного проекта: ECS

Обоснование:
1. **Зрелая экосистема** — документированные маппинги и правила детекции
2. **Интеграция Zeek** — существующие Elastic интеграции
3. **Знакомство аналитиков** — широко известна в SOC
4. **Практичность** — прямой путь к визуализации в Kibana

## 5. Реализация маппинга ECS

### 5.1 Таблица маппинга полей

| Zeek поле | Нормализованное | ECS поле | Тип |
|-----------|-----------------|----------|-----|
| `id.orig_h` | `src_ip` | `source.ip` | IP адрес |
| `id.resp_h` | `dst_ip` | `destination.ip` | IP адрес |
| `id.orig_p` | `src_port` | `source.port` | Целое (0-65535) |
| `id.resp_p` | `dst_port` | `destination.port` | Целое (0-65535) |
| `proto` | `protocol` | `network.transport` | Ключевое слово |
| `ts` | `timestamp` | `@timestamp` | ISO 8601 UTC |
| `uid` | `uid` | `event.id` | Ключевое слово |
| `service` | `service` | `network.protocol` | Ключевое слово |
| `duration` | `duration_sec` | `event.duration` | Дробное |
| `orig_bytes` | `bytes_sent` | `source.bytes` | Длинное целое |
| `resp_bytes` | `bytes_recv` | `destination.bytes` | Длинное целое |
| `query` | `dns_query` | `dns.question.name` | Ключевое слово |
| `qtype_name` | `dns_qtype` | `dns.question.type` | Ключевое слово |
| `rcode_name` | `dns_rcode` | `dns.response_code` | Ключевое слово |
| — | `log_type` | `event.dataset` | Ключевое слово |
| — | `ti_match` | `threat.indicator.matched` | Логическое |

### 5.2 Автоматически добавляемые поля

| ECS поле | Значение | Описание |
|----------|----------|----------|
| `event.kind` | `"event"` | Тип события для сетевой телеметрии |
| `event.category` | `"network"` | Категория (определяется по log_type) |
| `event.type` | `"connection"` / `"protocol"` | Подтип события |

### 5.3 Пример преобразования

**До (нормализованная схема):**
```
timestamp               log_type  uid                   src_ip         src_port  dst_ip          dst_port  protocol
2025-11-03 08:12:34     conn      CYfOwn3KhUWXr2GnVe    192.168.1.100  52134     93.184.216.34   443       tcp
2025-11-03 08:12:34     dns       CbNCRo1MkFJPBcb3ai    192.168.1.100  52135     8.8.8.8         53        udp
```

**После (ECS схема):**
```
@timestamp                   event.kind  event.category  event.dataset  source.ip      source.port  destination.ip   destination.port  network.transport
2025-11-03T08:12:34.112000Z  event       network         conn           192.168.1.100  52134        93.184.216.34    443               tcp
2025-11-03T08:12:34.450000Z  event       network         dns            192.168.1.100  52135        8.8.8.8          53                udp
```

## 6. Исследование автоматизации

### 6.1 Elastic Agent / Filebeat модули

**Описание:** Сборщики данных Elastic с предварительно настроенными модулями.

| Характеристика | Описание |
|----------------|----------|
| Поддержка Zeek | Нативный модуль `zeek` для всех типов логов |
| Маппинг схемы | Автоматический маппинг в ECS встроен |
| Развёртывание | На основе агента, управление через Fleet |
| Обработка пропусков | Корректная обработка с null значениями |

**Пример конфигурации:**
```yaml
filebeat.modules:
- module: zeek
  conn:
    enabled: true
    var.paths: ["/opt/zeek/logs/current/conn.log"]
```

### 6.2 Logstash Pipelines (Grok)

**Описание:** Парсинг на основе паттернов с использованием Grok выражений.

| Характеристика | Описание |
|----------------|----------|
| Гибкость | Кастомные паттерны для любого формата |
| Производительность | На основе JVM, умеренная пропускная способность |
| Обработка ошибок | Тег `_grokparsefailure` для некорректных логов |
| Отсутствующие поля | Условное добавление через `if [field]` |

**Пример Grok паттерна:**
```ruby
filter {
  grok {
    match => { "message" => "%{IP:source.ip}:%{INT:source.port}" }
  }
  mutate {
    rename => { "src" => "source.ip" }
  }
}
```

### 6.3 Elasticsearch Ingest Pipelines

**Описание:** Обработка на стороне сервера внутри Elasticsearch.

| Характеристика | Описание |
|----------------|----------|
| Расположение | Выполняется в кластере Elasticsearch |
| Процессоры | Grok, JSON, Date, GeoIP, User Agent и др. |
| Производительность | Распределённая, масштабируется с кластером |
| Обработка ошибок | `on_failure` обработчики для каждого процессора |

**Пример конвейера:**
```json
{
  "processors": [
    {
      "rename": {
        "field": "id.orig_h",
        "target_field": "source.ip",
        "ignore_missing": true
      }
    }
  ]
}
```

### 6.4 Security Onion

**Описание:** Интегрированный дистрибутив безопасности с преднастроенным парсингом.

| Характеристика | Описание |
|----------------|----------|
| Стек | Zeek + Suricata + Elasticsearch + Kibana |
| Парсинг | Конвейеры Logstash включены |
| Схема | Выровнена с ECS с кастомными расширениями |

### 6.5 Обработка несовершенных логов

**Типичные проблемы:**
- Отсутствующие поля (не все соединения имеют `service`)
- Несоответствия типов (числовые поля как строки)
- Обрезанные данные
- Проблемы кодировки

**Лучшие практики:**
```python
# 1. Значения по умолчанию
df["source.bytes"] = df["source.bytes"].fillna(0)

# 2. Приведение типов с обработкой ошибок
df["source.port"] = pd.to_numeric(df["source.port"], errors="coerce")

# 3. Предупреждения (не ошибки)
if df["source.ip"].isna().sum() > 0:
    logger.warning(f"Отсутствует source.ip в {count} строках")

# 4. Метрики покрытия
coverage = mapped_fields / total_fields * 100
```

## 7. Реализация автоматизации

```python
class ECSMapper:
    """Маппер Zeek/нормализованных логов в ECS."""

    def __init__(
        self,
        strict_validation: bool = False,  # Ошибка при проблемах валидации
        add_defaults: bool = True,         # Добавлять поля по умолчанию
        warn_unmapped: bool = True,        # Предупреждать о незамапленных
    ):
        ...

    def transform(
        self,
        df: pd.DataFrame,
        source_type: str = "normalized",
    ) -> tuple[pd.DataFrame, MappingReport]:
        """Преобразовать DataFrame в схему ECS."""
        ...
```

**Функции валидации:**

| Функция | Проверка |
|---------|----------|
| `validate_ip()` | Формат IPv4/IPv6 |
| `validate_port()` | Диапазон 0-65535 |
| `validate_timestamp()` | ISO 8601 или Unix epoch |
| `validate_keyword()` | Длина строки |
